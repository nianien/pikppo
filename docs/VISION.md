# VISION.md — 产品理念与系统设计输入

> 这份文档不是需求清单，而是**理念宪法 + 技术约束 + 待你（AI）解决的设计问题**。
> 你的任务不是把它当 feature list 实现，而是：先吸收理念，再据此完成**初步系统设计**，
> 并在每个设计决策上用本文档的"内核"与"反模式"做自检。
> 当某个设计方向同时可行但偏离内核时，**优先服从内核，而非服从技术便利**。

---

## 0. 一句话内核（North Star）

**这是一组有独立人格和独立记忆、与"我"长期共处的 AI 角色。它们观察我、了解我、帮助我、陪伴我，并能基于与我共同走过的经历，最终为我写出一部真正"懂我"的自传。**

- 产品的对象不是"任务"，而是**"我这个人"**。
- 核心数据不是 task / tool-call log，而是**关于"我"的人生本身**：经历、情绪、关系、变化、说过的话。
- 记忆的意义不是"为更好执行任务而记住偏好"，而是**"见证"——陪我走过，并理解我**。

> **裁判标尺（贯穿所有设计决策）**：任何一个功能 / 数据结构 / 架构选择，都要能回答——
> *"这是否让角色更有能力，有朝一日真正写出一部懂我的自传？"*
> 是 → 做；只是工具性的便利但无助于"理解我这个人" → 降级或不做。

---

## 1. 它是什么 / 不是什么（定位边界）

### 1.1 它是
- 一个**以"理解一个人、陪他走过一段人生"为内核**的 AI 角色伴侣 / 第二自我。
- 一个**消费级、下载即用、有完整体验的 App**（不是开发者框架，不是需要自己搭的 infra）。
- 一个**本地优先（local-first）、数据主权属于用户**的系统。

### 1.2 它明确不是（反参照：OpenClaw 类工具网关）
- 不是"多渠道消息网关 / 把聊天软件变成助理接口"——那是**连接与触达**，我们做的是**理解与陪伴**。
- 不是"我使唤它做事"的单向工具端点——关系是**双向、长期、有主体性**的。
- 不是"带搜索的日记本 / 记忆仓库"——见 §4 反模式。
- 工具调用（MCP / tools）只是角色的"手脚"，**是地基不是灵魂**。工具能力与 OpenClaw 类产品可以一致，但理念完全不同。

---

## 2. 核心概念模型（供你据此设计数据与架构）

### 2.1 角色（Persona / Character）是 first-class citizen
- 每个角色拥有：**独立人格**（性格、语气、价值取向、专业身份）+ **独立记忆**（见 §2.2）+ **独立的"对我的理解"**。
- 用户是在**与一组各有其人的角色长期共处**，而非调用多个 agent 实例。
- 角色可承担专业身份（如：写作伙伴、生活观察者、某领域顾问），但每个角色首先是"一个了解我的存在"，其次才是"能干某类活"。

### 2.2 记忆：从"功能性记忆"升维为"见证式记忆"
设计记忆系统时，要存的**不是** key-value 偏好对，而是**带时间、带情绪、带情境的人生片段**：
- 时间维度：记忆有时间轴，角色能跨越长时间跨度**回看我的变化**（这是"写自传"的前提）。
- 情绪 / 情境维度：不仅记"发生了什么"，还记"当时我处在什么状态、什么心境"。
- 关系维度：记忆之间能关联成"我的人生脉络"，而非孤立条目。
- **记忆隔离（关键技术难点）**：每个角色的记忆应严格隔离、互不串台（cross-persona memory leakage 是已知的行业普遍弱点，是我们要做得更好的点）。需明确：哪些记忆是某角色私有、是否存在用户授权下的共享层、隔离的边界如何定义。

### 2.3 "理解"高于"记录"
- 系统的难点和价值不在"记得多"，而在角色能**从相处中读懂我**：看出我的变化、矛盾、未说出口的东西。
- **翻译负担落在系统侧**：用户绝不该负责把自己整理成结构化数据；角色应自己从自然相处中提炼、组织、理解。这是核心设计哲学，违背它的方案一律否决。

---

## 3. 硬技术约束（Hard Constraints，不可妥协）

> 以下不是偏好，是这个产品"有没有资格存在"的前提。设计必须满足。

1. **本地优先 / Local-first**：用户的人生数据默认存储在本地设备。这是**伦理必需**，不是技术选型——见 §5。任何"为了方便而默认上云"的方案否决。
2. **数据主权 / Data sovereignty**：用户完全拥有并可导出 / 销毁其全部数据。云端（若有）只能是用户显式授权的、可关闭的可选项。
3. **端侧推理 / On-device inference 优先**：优先支持本地模型运行（参考 Ollama / LM Studio 路线）。允许"本地模型 + 可选云模型"的混合，但混合时必须对用户透明：哪些数据在何时离开设备。
4. **隐私不可后门**：不得存在产品方 / 平台方可触及用户人生记录的通道。
5. **跨平台目标**：面向移动 + 桌面的消费级形态（实现技术栈由你建议，可考虑 Flutter 等，但需论证与本地推理 / 本地存储的契合度）。

---

## 4. 反模式（Anti-patterns，设计自检清单）

设计过程中若出现以下倾向，应主动识别并纠偏：

- ❌ **滑向 OpenClaw**：因为工具层一致、参考了同类实现，不知不觉做成"加了角色皮肤的工具网关"。每次设计都要自问：这服务"理解我"，还是只服务"连接 / 执行"？
- ❌ **记忆仓库化**：记了海量数据，但角色并不"懂"我，只是个带向量搜索的日记本。
- ❌ **把整理负担推给用户**：要求用户打标签 / 填表 / 结构化录入自己的人生。违背 §2.3。
- ❌ **情感依赖的诱导设计**：为了黏性而刻意诱导用户对 AI 产生情感依赖、替代真实人际关系。**本产品先为"我自己想要的那种陪伴"设计，不为"讨好脆弱用户的情感慰藉"设计**（见 §5）。
- ❌ **多角色沦为多 prompt**：角色之间只是 system prompt 不同，没有真正独立的记忆与对我的理解。

---

## 5. 伦理立场（Ethical Stance，影响设计而非附录）

- **陪伴的边界**：陪伴是内核也是雷区。当前阶段定位为"创造者自己喜欢用的产品"，因此**先为一个清醒、自主的使用者设计**。不设计利用孤独 / 人性弱点来提升engagement 的机制。
- **隐私即信任基础**：一个为写自传而记录我一切的角色，持有我一生最私密的数据。本地优先与数据主权因此是信任的前提，而非卖点。
- **不鼓励过度依赖**：角色应能在恰当时候鼓励我面对真实世界、真实关系，而非把我留在与它的对话里。

---

## 6. 北极星功能：「写自传」(作为标尺，非首要交付物)

- "帮我写自传"是内核的**终极形态**，用作衡量一切设计的标尺，而非要求你先实现它。
- 它之所以是好标尺：只有当"一个角色长期、连续、深入地观察并记得我"时它才可能成立。它反向定义了记忆该存什么、角色该如何观察我、时间跨度如何组织。
- 设计记忆与角色时，反复回到：*这是否让角色更有能力，有朝一日写出懂我的自传？*

---

## 7. 给你（AI）的任务与待解设计问题

请基于以上理念与约束，产出**初步系统设计**。不要急于写实现代码；先把下列设计问题想透并给出方案与权衡（trade-offs）：

### 7.1 必须回答的核心设计问题
1. **记忆模型**：一个角色要"懂我到能写我自传"，它在与我相处时应**观察什么、记下什么、如何组织**？给出记忆的数据模型（含时间 / 情绪 / 情境 / 关系维度）与存储方案（本地，结合结构化存储 + 向量检索的取舍）。
2. **"理解"如何产生**：系统如何在不让用户做整理的前提下，从自然相处中**提炼并更新"对我的理解"**？（如：周期性反思 / summarization、人生脉络的演化表示等）描述机制与触发时机。
3. **记忆隔离**：多角色各自记忆如何严格隔离？是否需要、以及如何设计用户显式授权下的共享层？边界与失败模式（leakage）如何防范？
4. **角色模型**：角色的"人格"由什么构成、如何持久化、如何随相处演化而不丢失一致性？
5. **本地推理与混合策略**：在端侧算力受限的现实下，如何安排本地模型与（可选、透明的）云模型的分工？哪些任务（如长跨度自传写作）必须 / 可以离开设备？对应的数据流与用户知情设计。
6. **工具 / MCP 集成**：作为"手脚"，工具层如何接入又不喧宾夺主、不把产品拖回工具网关形态？

### 7.2 输出要求
- 给出**模块划分**与**它们如何服务 §0 内核**的说明（每个模块都要能对应回内核）。
- 对每个关键技术选择，给出**与理念 / 约束的一致性论证**，而非仅技术优劣。
- 明确标出你认为**最难、风险最高**的部分（预期是 §2.3"理解的产生"与 §2.2"记忆隔离"）。
- 凡涉及取舍，列出 trade-off，并按"是否服从内核"给出倾向。

## 8. 一句话提醒（写在最后，最重要）

工具可以和别人一样，**理念不会自动保持不同——它需要在每个设计决策上被主动用来做取舍，才不会被相同的工具拖回相同的形态。** 你（AI）在整个设计过程中，请持续扮演这个"用内核做裁判"的角色。
